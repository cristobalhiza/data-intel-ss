# Central de Datos de Empresas Chilenas: Guía de Enriquecimiento para RevOps

**TL;DR:** Este informe documenta **15 fuentes de datos públicos y gratuitos** para enriquecer información de empresas en Chile, con foco en resolver el problema clave de **obtener el dominio web a partir de un RUT o razón social**. La estrategia prioriza 5 fuentes de alta disponibilidad — `ruts.info`, `sre.cl`, datos del SII, NIC Chile WHOIS, y scraping de directorios — para construir un pipeline de enriquecimiento escalable que alimenta una base de datos central en **MariaDB**. El flujo completo, desde un RUT de input hasta un registro enriquecido con dominio, actividad económica, tamaño y otras variables comerciales, puede operar sin costos de licenciamiento y cumpliendo con el marco legal chileno de datos públicos.

---

## 1. La Arquitectura de la Solución

La construcción de una central de datos de empresas chilenas demanda una arquitectura que integre múltiples fuentes heterogéneas, cada una con sus propios formatos de salida, mecanismos de acceso y frecuencias de actualización. El diseño propuesto se articula en cuatro capas funcionales que transforman datos crudos en información estructurada y lista para consumo por herramientas de RevOps. Esta estructura modular permite incorporar nuevas fuentes sin alterar el núcleo del sistema, lo cual es crítico en un ecosistema donde los portales gubernamentales pueden cambiar sus interfaces sin previo aviso.

La primera capa, **Fuentes Primarias**, agrupa los sistemas oficiales que actúan como "sistemas de registro" — el SII, el Registro de Empresas y Sociedades, el Diario Oficial, NIC Chile, ChileCompra, INAPI y otras instituciones que generan datos primarios sobre la existencia legal y operativa de las empresas. La segunda capa, **Métodos de Extracción**, define cómo se accede a cada fuente: APIs públicas directas, scraping ético con BeautifulSoup o Scrapy, descarga de datasets abiertos en CSV/JSON, consultas DNS/WHOIS, búsquedas con Google Dorks y técnicas de matching con LinkedIn. La tercera capa, **Procesamiento**, aplica transformaciones de normalización (direcciones, nombres), validación (RUT con módulo 11), deduplicación, enriquecimiento cruzado entre fuentes y clasificación por tamaño. La cuarta capa, la **Base de Datos Central en MariaDB**, persiste el resultado final en tablas normalizadas que alimentan los sistemas downstream.

![Arquitectura Central de Datos](arquitectura_central_datos.png)

La decisión de usar **MariaDB** como destino responde a tres factores: su compatibilidad nativa con el ecosistema de datos abiertos del Estado chileno (que frecuentemente publica en formatos que se ingestan fácilmente en SQL), su capacidad para manejar volúmenes de millones de registros sin licenciamiento (crítico para una base de datos que puede crecer desde cientos de miles hasta varios millones de empresas), y la madurez de sus herramientas de replicación y backup. Para una implementación inicial, se recomienda una estructura de tablas que separa entidades principales (`empresas`, `dominios`, `actividades_economicas`, `documentos`) de tablas de enriquecimiento (`contactos`, `historial_constituciones`, `participaciones_licitaciones`), lo que permite consultas eficientes y actualizaciones incrementales sin bloquear tablas principales.

## 2. Fuentes Primarias: El Sistema Tributario y de Registro

### 2.1 Servicio de Impuestos Internos (SII)

El **SII** constituye la fuente de datos más autoritativa sobre empresas chilenas, ya que todo contribuyente que inicia actividades comerciales debe registrarse obligatoriamente ante este organismo. La información que el SII pone a disposición pública, accesible a través del portal de **"Situación Tributaria de Terceros"** y el repositorio de datos abiertos en `datos.gob.cl`, incluye el RUT, la razón social, los códigos de actividad económica (ACTECOs), la dirección comercial, la comuna, la fecha de inicio de actividades y la vigencia del inicio  [(ChileAtiende)](https://www.chileatiende.gob.cl/fichas/3130-situacion-tributaria-de-terceros) . Esta información es suficiente para construir el esqueleto básico de un registro empresarial: identidad única (RUT), nombre legal y ubicación geográfica. Los ACTECOs son particularmente valiosos porque utilizan la clasificación internacional CIIU, lo que permite segmentar empresas por industria con un estándar reconocido globalmente.

El acceso a los datos del SII presenta una restricción técnica relevante: **no existe una API pública oficial que permita consultar masivamente datos de contribuyentes sin autenticación**. La herramienta web de "Situación Tributaria de Terceros" requiere interactuar con formularios web, lo que implica que la extracción automatizada debe recurrir a técnicas de scraping o a intermediarios que ya han resuelto este problema. Es aquí donde entran en juego servicios como `ruts.info` y `sre.cl`, que actúan como capas de abstracción sobre los datos del SII, ofreciendo interfaces API más amigables para desarrolladores. El Servicio de Impuestos Internos también publica datasets de **constituciones de empresas** en `datos.gob.cl` con periodicidad mensual, incluyendo el RUT, razón social, fecha de constitución y domicilio — un recurso fundamental para mantener la base actualizada con nuevas empresas  [(Portal de Datos Abiertos)](https://datos.gob.cl/dataset/registro-de-empresas-y-sociedades/resource/71c8e355-226a-461e-809a-870c2275a178?inner_span=True) .

### 2.2 ruts.info: API Gratuita sin Autenticación

**ruts.info** representa uno de los hallazgos más valiosos de esta investigación por su simplicidad de acceso y la riqueza de datos que devuelve. Desarrollada como respuesta a la ausencia de APIs públicas gratuitas para datos de empresas chilenas, esta API permite consultar información a partir de un RUT sin necesidad de registro, API key o autenticación de ningún tipo  [(Github)](https://github.com/juanbrujo/listado-apis-publicas-en-chile/issues/71) . El endpoint `https://ruts.info/api/company-info?rut={rut}` responde en formato JSON y devuelve campos que incluyen **razón social, nombre de fantasía, giro comercial, dirección, comuna, dominio web, email y otros datos de contacto**. La presencia del campo **dominio web** en la respuesta resuelve directamente uno de los requerimientos centrales del proyecto, permitiendo enriquecer un registro empresarial con su sitio web asociado en una sola llamada HTTP.

La API es mantenida como un proyecto comunitario y, si bien no cuenta con SLA ni garantías de uptime, su modelo de acceso libre la convierte en el punto de partida ideal para un pipeline de enriquecimiento. La recomendación técnica es implementar un sistema de **cache local** que almacene las respuestas exitosas para evitar consultas repetidas al mismo RUT, reduciendo la carga sobre el servicio y acelerando los tiempos de procesamiento batch. Para un procesamiento masivo de miles de RUTs, se sugiere implementar rate limiting del lado del cliente (por ejemplo, 1-2 segundos entre peticiones) como medida de cortesía y para evitar ser bloqueado.

### 2.3 sre.cl: API Pública para Consulta Masiva

El **Servicio de Registro de Empresas (sre.cl)** ofrece una API pública diseñada específicamente para integraciones con sistemas de facturación electrónica, que resulta igualmente útil para propósitos de enriquecimiento de datos comerciales  [(SRE)](https://sre.cl/) . La API opera sobre el endpoint `https://sre.cl/api/company_info` y acepta un token público (`token_publico`) que no requiere registro. Su principal ventaja competitiva frente a ruts.info es la capacidad de **consulta masiva**: permite enviar hasta **100 RUTs en una sola petición** mediante el parámetro `ruts`, recibiendo un objeto JSON donde cada clave es un RUT y su valor es el conjunto de datos asociado. Esto reduce drásticamente el tiempo de procesamiento batch, ya que una tanda de 10,000 empresas requiere solo 100 llamadas API en lugar de 10,000.

Los datos devueltos por sre.cl incluyen razón social, actividades económicas (con descripción y código), régimen tributario, dirección completa y datos de contacto. La API también implementa un mecanismo de **checksum** (`check_ruts`) que permite verificar cuáles registros han cambiado desde una fecha determinada, optimizando los procesos de actualización incremental de la base de datos central. Para el caso de uso de RevOps, donde la prioridad es obtener dominios web, sre.cl se posiciona como complemento de ruts.info: mientras ruts.info puede devolver el dominio directamente, sre.cl entrega información más detallada sobre actividades económicas que facilita la segmentación por industria.

## 3. Descubrimiento de Dominios Web

### 3.1 NIC Chile: WHOIS y Listados de Dominios

**NIC Chile**, operado por la Universidad de Chile, es la entidad encargada de administrar el dominio de nivel superior **.cl**. Esta institución pone a disposición pública dos herramientas de alto valor para el descubrimiento de dominios empresariales: el servicio de **consulta WHOIS** y los **listados CSV de dominios registrados y eliminados**  [(NIC Chile)](https://nic.cl/) . La consulta WHOIS permite buscar información sobre un dominio específico inscrito en .cl, incluyendo el nombre del titular, fecha de registro, fecha de expiración, servidores DNS y datos de contacto administrativo. Si bien el WHOIS no establece una relación directa entre un RUT y un dominio (el titular puede ser una persona natural o los datos pueden estar protegidos por privacidad), sí permite validar si un dominio candidato está efectivamente registrado y quién lo administra.

Los **listados CSV** constituyen una fuente aún más poderosa para el descubrimiento a gran escala. NIC Chile publica periódicamente archivos con los últimos dominios inscritos en formato CSV, disponibles por hora, día, semana y mes  [(NIC Chile)](https://www.nic.cl/dominios/) . Estos archivos contienen el nombre de dominio y la fecha de inscripción, lo que permite construir un inventario completo de dominios .cl existentes — más de **744,000 dominios registrados** al momento de esta investigación  [(NIC Chile)](https://nic.cl/) . La estrategia de enriquecimiento consiste en cruzar los nombres de dominio con las razones sociales de la base de datos de empresas: si una empresa llamada "Constructora XYZ SpA" tiene un dominio "constructoraxyz.cl", los algoritmos de matching por similitud de cadenas (Levenshtein distance, fuzzy matching) pueden establecer la asociación. Este enfoque, aunque requiere procesamiento, tiene la ventaja de no depender de que la empresa haya declarado su dominio en ninguna base de datos oficial.

### 3.2 Scraping de Directorios Empresariales

Los directorios empresariales chilenos representan una fuente rica de datos de contacto y dominios web, ya que su modelo de negocio depende de que las empresas sean encontradas por clientes potenciales. **Amarillas.com** (del Grupo El Mercurio), **EncuentraEmpresas.cl** y otros directorios similares acumulan miles de fichas empresariales con información que incluye nombre comercial, dirección, teléfono, sitio web y categoría de actividad  [(EncuentraEmpresas.cl)](https://www.encuentraempresas.cl/empresa-amarillas-com-una-empresa-del-grupo-el-mercurio.html) . EncuentraEmpresas.cl destaca por indicar explícitamente que sus datos provienen del **SII** bajo la Ley 19.628 de Protección de Datos Personales, lo que confiere una capa adicional de legitimidad a la información  [(EncuentraEmpresas.cl)](https://www.encuentraempresas.cl/empresa-amarillas-com-una-empresa-del-grupo-el-mercurio.html) .

El scraping de estos directorios debe realizarse bajo estrictos parámetros éticos y legales. La investigación revela que el web scraping en Chile es legal cuando se cumplen ciertas condiciones: los datos deben ser de acceso público (sin autenticación ni restricciones), no se debe eludir medidas de seguridad como CAPTCHAs o firewalls, y se debe respetar el archivo `robots.txt` de cada sitio  [(flunt.cl)](https://flunt.cl/2025/03/05/web-scraping-legal-chile-empresas/) . Las mejores prácticas técnicas incluyen implementar rate limiting (máximo 1-2 peticiones por segundo), usar un `User-Agent` identificable con información de contacto, y almacenar los datos de forma temporal para evitar acumulación innecesaria  [(DataCamp)](https://www.datacamp.com/es/blog/ethical-web-scraping) . Desde el punto de vista técnico, herramientas como **BeautifulSoup** con la librería `requests` de Python son suficientes para la mayoría de los casos, mientras que **Scrapy** ofrece mayor robustez para extracciones a gran escala con manejo de reintentos, rotación de proxies y exportación directa a múltiples formatos  [(SerpApi)](https://serpapi.com/blog/beautiful-soup-build-a-web-scraper-with-python/) .

### 3.3 Google Dorks y Búsqueda Web

Las **técnicas de búsqueda avanzada** en Google, conocidas como "Google Dorks", constituyen un método complementario para descubrir dominios web cuando las fuentes estructuradas no arrojan resultados. La lógica es sencilla: si una empresa chilena tiene un sitio web, es muy probable que Google lo haya indexado. Operadores de búsqueda como `site:.cl "razón social"`, `intitle:"razón social" Chile`, o búsquedas que combinen el RUT con términos como "contacto" o "sitio web" pueden revelar dominios que no aparecen en bases de datos oficiales. Esta técnica es particularmente efectiva para empresas medianas y grandes que mantienen presencia web activa.

La implementación automatizada de Google Dorks requiere precauciones técnicas: Google aplica límites de rate agresivos (bloqueos temporales después de un número reducido de consultas automatizadas) y puede presentar CAPTCHAs. La solución recomendada es utilizar la **Custom Search JSON API** de Google, que ofrece 100 consultas gratuitas por día (con posibilidad de ampliar), o alternativas como **SerpAPI** que abstraen la complejidad de la extracción. Para casos donde el presupuesto es cero, una estrategia híbrida funciona bien: usar las APIs gratuitas para los casos más prioritarios y reservar el scraping directo de resultados Google para lotes pequeños de empresas de alto valor donde el dominio web no ha sido encontrado por otros medios.

### 3.4 LinkedIn: Matching de Empresas

**LinkedIn** alberga perfiles de empresa para la gran mayoría de las organizaciones chilenas con presencia profesional, y cada perfil de empresa incluye —cuando está disponible— la URL del sitio web oficial. La plataforma **Datablist** ha documentado una técnica de "database matching" que permite obtener la URL web de una empresa a partir de su página de LinkedIn sin necesidad de scraping costoso, consultando una base de datos que mapea dominios con páginas de empresa en LinkedIn  [(Datablist)](https://www.datablist.com/es/how-to/urls-de-empresa-de-linkedin-a-sitios-web) . Para el caso chileno, esta técnica puede adaptarse: si se conoce el nombre de la empresa y se encuentra su perfil de LinkedIn, el campo "Website" del perfil de empresa proporciona el dominio buscado.

El scraping directo de LinkedIn presenta mayores restricciones técnicas que otros sitios: la plataforma detecta patrones de acceso automatizado con alta eficacia y puede suspender cuentas. La alternativa más segura es usar la **API oficial de LinkedIn** (disponible para partners de Microsoft) o herramientas de terceros que operan con infraestructura propia como **Vayne** o **Phantombuster**  [(vayne.io)](https://www.vayne.io/en/linkedin-profile-company-scraper) . Para un proyecto sin presupuesto de licenciamiento, la estrategia recomendada es usar LinkedIn como fuente de verificación manual para un subconjunto priorizado de empresas, más que como fuente de extracción masiva.

## 4. Fuentes Complementarias de Enriquecimiento

### 4.1 Registro de Empresas y Sociedades (RES) y Diario Oficial

El **Registro de Empresas y Sociedades (RES)**, accesible a través de `www.registrodeempresasysociedades.cl` y `tuempresaenundia.cl`, es el sistema digital del Ministerio de Economía mediante el cual se constituyen, modifican y disuelven sociedades comerciales en Chile. Desde su lanzamiento en mayo de 2013, el RES ha acumulado más de **1.46 millones de constituciones**, representando actualmente el **91.7%** del total de empresas constituidas (el restante 8.3% se registra por el sistema tradicional del Diario Oficial)  [(Ministerio de Economía, Fomento y Turismo)](https://www.economia.gob.cl/wp-content/uploads/2025/12/informe-res-2025-noviembre.pdf) . El portal público permite búsquedas por RUT, razón social, fecha de actuación o fecha de registro **sin necesidad de iniciar sesión**  [(REGISTRO DE EMPRESAS Y SOCIEDADES)](https://www.registrodeempresasysociedades.cl/BuscarActuaciones.aspx) , lo que facilita la extracción automatizada de datos de constitución.

Los datasets del RES se publican mensualmente en `datos.gob.cl` en formatos CSV, TSV, JSON y XML, e incluyen el RUT de la sociedad, razón social, tipo de sociedad (SpA, SRL, EIRL, SA), domicilio, fecha de constitución y datos de los socios  [(Portal de Datos Abiertos)](https://datos.gob.cl/dataset/registro-de-empresas-y-sociedades/resource/71c8e355-226a-461e-809a-870c2275a178?inner_span=True) . Estos datasets son especialmente valiosos para dos casos de uso: mantener la base de datos actualizada con **nuevas empresas constituidas** cada mes, y enriquecer los registros existentes con información sobre el tipo societario y la antigüedad de la empresa. El **Diario Oficial** complementa esta información con las publicaciones de constitución, modificación y disolución de sociedades que operan bajo el régimen tradicional  [(Diario Oficial)](https://www.diariooficial.interior.gob.cl/tramites/sociedades/conocer/) , accesibles a través del portal de sociedades con CVE (Código de Validación Electrónica).

### 4.2 ChileCompra y Mercado Público

La **Dirección de Compras y Contratación Pública (ChileCompra)** administra la plataforma **Mercado Público**, a través de cual el Estado de Chile adquiere bienes y servicios por montos que superan los **US$10.000 millones anuales**. La información de todas las licitaciones, órdenes de compra y adjudicaciones es de carácter público y se expone a través de la **API OCDS** (Open Contracting Data Standard), disponible en `api.mercadopublico.cl` y `datos-abiertos.chilecompra.cl`  [(ChileCompra)](https://www.chilecompra.cl/api/) . Los datos cubren el período desde 2022 en adelante y se actualizan en tiempo real, conteniendo más de **4.8 millones de licitaciones** y **4.9 millones de adjudicaciones** con más de **11.4 millones de partes involucradas**  [(open-contracting.org)](https://data.open-contracting.org/es/publication/144) .

Para propósitos de enriquecimiento de datos empresariales, la API de ChileCompra permite identificar **qué empresas son proveedoras del Estado**, en qué categorías de productos/servicios operan, qué montos han facturado y a qué organismos públicos han vendido. Esta información es un indicador de solvencia y estabilidad comercial extremadamente valioso para equipos de RevOps, ya que una empresa que vende consistentemente al Estado durante múltiples años demuestra capacidad operativa y cumplimiento de requisitos legales y tributarios. La API no requiere autenticación para consultas básicas y permite descargar datos en formato JSON, Excel o CSV  [(Mercado Público - Datos abiertos)](https://datos-abiertos.chilecompra.cl/) . Es importante notar que la nueva **API de Compra Ágil**, que cubre compras de hasta 100 UTM, está en desarrollo y se espera su lanzamiento en versión Beta durante 2026  [(ChileCompra)](https://www.chilecompra.cl/2026/04/chilecompra-lanzara-en-mayo-nueva-api-de-compra-agil-tras-proceso-de-consulta-publica/) , lo que ampliará significativamente la cobertura de datos.

### 4.3 INAPI: Propiedad Industrial como Indicador de Innovación

El **Instituto Nacional de Propiedad Industrial (INAPI)** pone a disposición pública sus bases de datos de **marcas y patentes** solicitadas y registradas a través del portal de datos abiertos  [(Inapi)](https://www.inapi.cl/datos-abiertos) . Para cada solicitud de marca o patente, los datos incluyen el nombre del solicitante (que puede ser una persona natural o jurídica), la dirección, el tipo de signo o invención, la clasificación internacional (Clasificación de Niza para marcas, Clasificación IPC para patentes) y el estado del trámite. La búsqueda se realiza a través del buscador de marcas en `buscadormarcas.inapi.cl` o el buscador de patentes en las secciones correspondientes del sitio web  [(inapi.cl)](https://buscadormarcas.inapi.cl/) .

El valor de INAPI como fuente de enriquecimiento radica en que una empresa que ha registrado marcas o patentes revela información sobre su estrategia de negocio: las marcas indican la estructura de portafolio de productos/servicios, mientras que las patentes revelan inversión en I+D y capacidad tecnológica. Para una base de datos de RevOps, el flag "tiene marca registrada" o "tiene patente" puede usarse como proxy de **madurez empresarial** y **innovación**, segmentando prospectos por nivel de sofisticación comercial. La extracción de datos de INAPI se realiza a través de su interfaz web, donde los resultados de búsqueda se pueden paginar y procesar con herramientas de scraping. Los datasets completos también están disponibles para descarga masiva en la sección de datos abiertos del sitio.

### 4.4 CMF Chile y Datos Financieros

La **Comisión para el Mercado Financiero (CMF)**, sucesora de la antigua SBIF, regula y supervisa las entidades del sistema financiero chileno. A través de su portal en `www.cmfchile.cl` y las APIs `api.sbif.cl` y `api.cmfchile.cl`, pone a disposición datos sobre bancos, administradoras de fondos de pensiones, compañías de seguros, fondos mutuos y otras entidades supervisadas  [(cmfchile.cl)](https://www.cmfchile.cl/portal/principal/613/w3-propertyvalue-29006.html) . La **API CMF Bancos** (versión 3.0) ofrece acceso a diversos recursos de información con datos actuales e históricos de diferentes tipos de reportes que las entidades financieras deben presentar periódicamente  [(sbif.cl)](https://api.sbif.cl/documentacion/index.html) .

Para el enriquecimiento de datos empresariales, la CMF aporta información valiosa principalmente cuando la empresa objetivo es una **entidad financiera regulada** (bancos, cooperativas, administradoras). Sin embargo, la CMF también publica datos del mercado de capitales que incluyen empresas **cotizadas en la Bolsa de Santiago**, con información sobre accionistas, hechos esenciales, estados financieros y indicadores de gobierno corporativo  [(cmfchile.cl)](https://www.cmfchile.cl/institucional/mercados/entidad.php?mercado=V&rut=90249000&grupo=&tipoentidad=RVEMI&row=&vig=VI&control=svs&pestania=21) . La Bolsa de Santiago, por su parte, ofrece su propia API de "Servicios de Consulta" con datos de instrumentos, transacciones, índices y profundidad de mercado  [(Santiago Stock Exchange)](https://www.bolsadesantiago.com/) , que puede usarse para enriquecer perfiles de empresas cotizadas con datos de capitalización bursátil y volumen de negociación.

### 4.5 Banco Central: Estadísticas Empresariales

El **Banco Central de Chile** mantiene la **Base de Datos Estadísticos (BDE)** con una sección de "Estadísticas Experimentales" que incluye datos sobre la **dinámica empresarial** clasificada por tamaño de empresa  [(Banco Central de Chile)](https://si3.bcentral.cl/Siete/ES/Siete/Cuadro/CAP_ESTADIST_EXPERIM/MN_EXPERIM01/EST_EXP_DEA_02/638667365404573244) . Las categorías utilizadas por el Banco Central son: pequeña empresa (ventas hasta UF 25,000), mediana empresa (ventas hasta UF 100,000) y gran empresa (ventas sobre UF 100,000). Estos datos son agregados y no identifican empresas individuales, pero sirven como benchmark para clasificar una empresa según sus ventas declaradas, comparándolas contra los umbrales oficiales del banco emisor.

La **API del BDE** permite acceder programáticamente a series de tiempo de múltiples indicadores, y el servicio open source **mindicador.cl** ofrece una API REST simplificada que entrega los principales indicadores económicos diarios para Chile en formato JSON, incluyendo UF, dólar, euro, IPC, IMACEC y otros  [(Mindicador)](https://mindicador.cl/) . Aunque estos indicadores son macroeconómicos y no datos de empresa individual, son útiles para contextualizar el entorno comercial en el que opera una empresa y para calcular métricas ajustadas por inflación. Por ejemplo, el valor de contratos adjudicados por ChileCompra puede convertirse a UF para comparaciones consistentes en el tiempo.

## 6. Pipeline de Enriquecimiento: De RUT a Registro Completo

### 6.1 Flujo de Procesamiento Paso a Paso

El diagrama a continuación ilustra el flujo completo de enriquecimiento, desde un RUT de entrada hasta un registro empresarial completamente enriquecido en la base de datos MariaDB. El proceso se organiza en **9 pasos secuenciales** que combinan APIs directas, técnicas de descubrimiento y procesamiento de datos.

![Flujo de Enriquecimiento](flujo_enriquecimiento.png)

El **Paso 1** recibe el input: un RUT, una razón social o incluso un nombre de fantasía. El **Paso 2** consulta inmediatamente la API de `ruts.info`, que en una sola llamada puede devolver razón social, giro, dirección y —críticamente— el dominio web. Si `ruts.info` no tiene el dominio, el **Paso 3** consulta `sre.cl` para obtener actividades económicas detalladas y datos de dirección. El **Paso 4** persiste estos datos básicos en MariaDB en la tabla `empresas`.

El **Paso 5** es el de descubrimiento activo del dominio: si el dominio no se obtuvo en los pasos anteriores, se activan múltiples estrategias en paralelo. La **NIC Chile WHOIS** permite buscar dominios que contengan la razón social; los **Google Dorks** indexan la web pública buscando sitios que mencionen la empresa; los **directorios empresariales** pueden tener la ficha con el sitio web; y **LinkedIn** ofrece el campo website en los perfiles de empresa. El **Paso 6** enriquece el dominio encontrado con datos DNS (registros MX para inferir infraestructura de email, registros A/AAAA para validar que el sitio responde, subdominios descubiertos, y certificados SSL).

El **Paso 7** consulta fuentes complementarias: **ChileCompra** verifica si la empresa es proveedora del Estado, **INAPI** verifica marcas y patentes, el **Diario Oficial** confirma la vigencia de la constitución, y el **Banco Central** aporta contexto de tamaño sectorial. El **Paso 8** aplica todo el procesamiento de normalización: validación del RUT con módulo 11, estandarización de direcciones (comuna, región, geocodificación), deduplicación de registros, enriquecimiento cruzado entre fuentes, clasificación por tamaño según ventas, y asignación del código CIIU más representativo. Finalmente, el **Paso 9** persiste el registro enriquecido en MariaDB con su score de completitud.

### 6.2 Estructura de Base de Datos Recomendada

La base de datos MariaDB debe diseñarse con tablas normalizadas que permitan ingestión incremental desde múltiples fuentes y consultas eficientes por cualquier dimensión de búsqueda. La tabla central `empresas` almacena los datos identitarios y de contacto, mientras que las tablas satélite desacoplan dimensiones que pueden tener múltiples valores o que provienen de fuentes con diferentes frecuencias de actualización.

| Tabla | Campos Clave | Fuente Principal | Actualización |
|---|---|---|---|
| `empresas` | id, rut, razon_social, nombre_fantasia, direccion, comuna, region, codigo_ciiu, tamano_empresa, fecha_inicio_actividades, score_completitud | SII, ruts.info, sre.cl | Mensual (batch) + on-demand |
| `dominios_web` | id, empresa_id, dominio, fuente_descubrimiento, fecha_registro_nic, dns_mx, ssl_vigente, fecha_descubrimiento | NIC Chile, ruts.info, scraping | Semanal |
| `actividades_economicas` | id, empresa_id, codigo_acteco, descripcion, es_principal, fuente | SII, sre.cl | Mensual |
| `contactos` | id, empresa_id, tipo (email/telefono), valor, fuente, verificado | ruts.info, directorios, scraping | Trimestral |
| `historial_constituciones` | id, empresa_id, tipo_actuacion, fecha, notario, folio, fuente | RES, Diario Oficial | Mensual (nuevas) |
| `participaciones_licitaciones` | id, empresa_id, id_licitacion, monto_adjudicado, fecha, organismo_comprador | ChileCompra OCDS | Semanal |
| `propiedad_industrial` | id, empresa_id, tipo (marca/patente), numero_solicitud, clasificacion, estado | INAPI | Trimestral |

El campo `score_completitud` en la tabla `empresas` es una métrica calculada que indica qué porcentaje de los campos deseables están poblados para cada registro, permitiendo priorizar qué empresas necesitan más esfuerzo de enriquecimiento. La tabla `dominios_web` incluye la trazabilidad del origen (`fuente_descubrimiento`) para poder auditar de dónde proviene cada dominio y regenerar el dato si la fuente original cambia.

## 7. Scraping Ético: Marco Legal y Buenas Prácticas

### 7.1 Legallidad del Scraping en Chile

La investigación confirma que el **web scraping es legal en Chile** cuando se operan dentro de ciertos límites bien definidos. El artículo de Flunt.cl especializado en la materia establece que la automatización de consultas en plataformas públicas puede ser legal siempre que la información sea de acceso público (sin autenticación ni restricciones), no se eludan medidas de seguridad como CAPTCHAs o firewalls, y se respetan los términos y condiciones de los sitios web  [(flunt.cl)](https://flunt.cl/2025/03/05/web-scraping-legal-chile-empresas/) . La **Ley 19.628 sobre Protección de la Vida Privada** protege los datos personales, pero no aplica a la información de empresas que es de carácter público por mandato legal. La **Ley 21.459 sobre Delitos Informáticos** penaliza el acceso indebido a sistemas computacionales, pero no contempla como delito el acceso automatizado a información públicamente disponible.

Desde una perspectiva ética, los principios fundamentales son la **transparencia técnica** (respetar `robots.txt`, usar `User-Agent` identificable), la **tasa de peticiones responsable** (rate limiting y backoff exponencial), y el **respeto a la privacidad** (no recolectar datos personales de individuos sin base legal)  [(DataCamp)](https://www.datacamp.com/es/blog/ethical-web-scraping) . El archivo `robots.txt` de un sitio web indica qué partes pueden ser rastreadas por robots, y aunque su incumplimiento no es necesariamente ilegal, constituye una señal clara de las intenciones del propietario del sitio que debe ser considerada  [(DataCamp)](https://www.datacamp.com/es/blog/ethical-web-scraping) .

### 7.2 Implementación Técnica Responsable

La implementación de scrapers debe seguir un checklist técnico que garantice operaciones sostenibles a largo plazo. El uso de **headers HTTP apropiados** es fundamental: el `User-Agent` debe identificar al bot con un nombre descriptivo y un correo de contacto, como `CompanyDataBot/1.0 (data-team@empresa.cl)`. Las **pausas entre peticiones** deben ser de al menos 1-2 segundos para no sobrecargar los servidores, y se debe implementar **backoff exponencial** cuando se reciban códigos de error 429 (Too Many Requests) o 503 (Service Unavailable)  [(Auto Cloud Labor - Dedica tu tiempo a lo que de verdad importa)](https://autclab.com/buenas-practicas-web-scraping/) .

La gestión de errores debe ser robusta: manejar explícitamente los códigos **403 Forbidden** (acceso denegado, detener el scraping), **404 Not Found** (registro inexistente, continuar con el siguiente), y **500 Internal Server Error** (reintentar con delay). Es recomendable usar **Scrapy** como framework principal porque incluye funcionalidades de reintentos automáticos, pipelines de exportación a múltiples formatos (JSON, CSV, XML), y un sistema de middlewares que permite rotar proxies y manejar sesiones de forma transparente  [(Bright Data)](https://brightdata.com/blog/how-tos/beautiful-soup-web-scraping) . Para sitios que requieren interacción con JavaScript, **Playwright** o **Selenium** permiten controlar un navegador real, aunque a costa de mayor consumo de recursos.

## 8. Estrategia de Implementación por Fases

### 8.1 Fase 1: Fundación (Semanas 1-3)

La primera fase se centra en establecer la infraestructura base y poblar la base de datos con los datos fundamentales. Se implementan las integraciones con las **5 fuentes de prioridad Alta**: `ruts.info`, `sre.cl`, SII vía `datos.gob.cl`, NIC Chile WHOIS/listados CSV, y scraping de directorios empresariales. El objetivo es tener una tabla `empresas` funcional con RUT, razón social, giro, dirección y —cuando sea posible— dominio web, cubriendo al menos **100,000 empresas** en el primer ciclo de procesamiento. La infraestructura de MariaDB se despliega con el esquema de tablas propuesto, incluyendo índices en RUT (único), razón social (búsqueda parcial) y dominio web.

El desarrollo del scraper para directorios empresariales debe comenzar por **EncuentraEmpresas.cl** como prioridad, ya que sus datos provienen del SII y tienen una estructura HTML relativamente consistente. Se debe implementar un sistema de logging que registre cada petición, su resultado y cualquier error, para facilitar la depuración y el monitoreo de la salud del pipeline. Al final de esta fase, el equipo de RevOps debe poder consultar la base de datos por RUT o nombre de empresa y obtener al menos los datos básicos de identificación y contacto.

### 8.2 Fase 2: Enriquecimiento (Semanas 4-6)

La segunda fase expande la cobertura de datos activando las fuentes de prioridad Media. Se integra la API OCDS de **ChileCompra** para identificar qué empresas son proveedoras del Estado y en qué categorías. Se implementa el scraping del **Registro de Empresas y Sociedades** para obtener datos de constitución y tipo societario. Se consulta **INAPI** para marcar qué empresas tienen propiedad intelectual registrada. Se implementa el módulo de **Google Dorks** como mecanismo de respaldo para descubrir dominios que no fueron encontrados por las fuentes de la Fase 1.

Durante esta fase también se desarrolla el módulo de **procesamiento y normalización**: validación de RUTs con módulo 11, geocodificación de direcciones (asignación de coordenadas latitud/longitud a partir de direcciones textuales), clasificación por tamaño según los umbrales del Banco Central, y cálculo del score de completitud de cada registro. Se implementa la deduplicación por fuzzy matching de razones sociales, ya que una misma empresa puede aparecer con ligeras variaciones en su nombre en diferentes fuentes.

### 8.3 Fase 3: Consolidación y Automatización (Semanas 7-8)

La tercera fase se enfoca en la automatización de los procesos de actualización y en la integración con los sistemas de consumo. Se configuran **jobs cron** o schedulers (como Apache Airflow o incluso cron simple) para ejecutar las extracciones de forma periódica: diaria para APIs rápidas (ruts.info, sre.cl), semanal para scraping de directorios, y mensual para datasets de datos.gob.cl. Se implementa un sistema de **alertas** que notifique cuando una fuente deja de responder o cuando la tasa de errores supera un umbral configurable.

La integración con los sistemas de RevOps (CRM, ERP, herramientas de BI) se realiza mediante vistas SQL o endpoints API que exponen los datos enriquecidos. Se recomienda crear una **vista materializada** que consolide toda la información por empresa en un solo registro, para que las herramientas de consumo no necesiten hacer joins complejos. Se documenta el proceso completo, incluyendo el origen de cada campo, la frecuencia de actualización y las limitaciones conocidas de cada fuente.

## 9. Consideraciones Técnicas y Limitaciones

### 9.1 Rate Limiting y Disponibilidad de Fuentes

Una limitación inherente a cualquier proyecto que dependa de fuentes públicas es la **inestabilidad de los servicios**. Los portales gubernamentales chilenes, aunque generalmente confiables, pueden experimentar mantenciones programadas, caídas por sobrecarga o cambios en sus interfaces sin previo aviso. La consulta ciudadana realizada por ChileCompra en marzo de 2026 identificó explícitamente la **inestabilidad del servicio** y los errores HTTP 500 como uno de los cinco problemas principales reportados por usuarios de sus APIs  [(ChileCompra)](http://www.chilecompra.cl/wp-content/uploads/2026/04/Resultados_ConsultaCiudadana_APIs.pdf) . La estrategia de mitigación es implementar **circuit breakers** que desactiven temporalmente una fuente cuando detectan múltiples fallos consecutivos, y **colas de reintentos** que reprogramen las consultas fallidas para más tarde.

El rate limiting del lado del cliente es igualmente importante. Aunque fuentes como `ruts.info` no documentan límites explícitos, la cortesía hacia los servidores públicos y la autopreservación del acceso exigen pausas razonables entre peticiones. Una configuración conservadora de **1 petición por segundo** para APIs, **1 petición cada 2-3 segundos** para scraping web, y **backoff exponencial** ante errores (1s, 2s, 4s, 8s...) proporciona un balance entre velocidad de procesamiento y sostenibilidad del acceso.

### 9.2 Calidad y Completitud de los Datos

La calidad de los datos de fuentes públicas varía significativamente. El SII tiene información sobre aproximadamente **2.5 millones de contribuyentes** con inicio de actividades, pero no todos son empresas comerciales activas: el registro incluye personas naturales con actividad independiente, sociedades inactivas, y empresas que cambiaron de razón social sin actualizar sus datos. El **porcentaje de empresas que tienen un dominio web registrado** en fuentes públicas es considerablemente menor al 100%: muchas microempresas y pymes chilenas operan sin presencia web formal, o usan redes sociales (Facebook, Instagram) como su único canal digital.

El campo **dominio web** tendrá naturalmente una tasa de cobertura inferior a campos obligatorios como RUT o razón social. Basado en la experiencia de plataformas similares, se estima que entre el **30% y 50%** de las empresas chilenas con inicio de actividades tendrán un dominio web asociado descubrible por las técnicas documentadas. Para las empresas restantes, el pipeline debe marcar el dominio como "no encontrado" y permitir la entrada manual o la reconsulta periódica, ya que una empresa puede crear su sitio web después de haber sido procesada.

### 9.3 Actualización y Mantención

Los datos empresariales son dinámicos: las empresas cambian de dirección, modifican su giro, se disuelven o fusionan. El pipeline debe implementar un mecanismo de **actualización incremental** que no reprocese desde cero toda la base en cada ciclo, sino que identifique qué registros han cambiado desde la última ejecución. La API de `sre.cl` ofrece el endpoint `check_ruts` que compara fechas de modificación, mientras que los datasets de `datos.gob.cl` incluyen timestamps de actualización que permiten filtrar solo los registros modificados  [(SRE)](https://sre.cl/) .

Para el scraping de directorios, la estrategia es rastrear el "hash" del contenido HTML de cada ficha empresarial: si el hash cambia entre visitas, la ficha ha sido actualizada y debe reprocesarse. Para NIC Chile, los listados CSV de dominios nuevos y eliminados permiten mantener el inventario de dominios actualizado sin necesidad de reescanear todo el namespace .cl. El objetivo de mantención es que el pipeline pueda ejecutarse incrementalmente en **menos de una hora** para un ciclo de actualización diaria, en lugar de las múltiples horas que requeriría un reprocesamiento completo.
